# 동문 사용자 언어 원칙

일반 동문 화면의 문구는 사용자가 다음 행동을 바로 이해할 수 있는 한국어를 사용합니다.
코드에서 반복 사용하는 핵심 명칭은 `apps/web/lib/member-language.ts`와 함께 관리합니다.

## 공통 용어

| 내부·기존 표현 | 일반 동문 화면 | 사용 원칙 |
| --- | --- | --- |
| 관리자 문의 | 동문회 사무국 문의 | 문의 주체를 실제 조직명으로 안내 |
| 신규 가입신청 | 동문 가입 신청 | 대상과 행동을 함께 표시 |
| 계정 활성화 | 첫 로그인 설정 | 사용자가 수행할 행동으로 설명 |
| token, 활성화 토큰 | 계정 활성화 코드 | 입력 화면에서만 사용하고 기술 형식은 숨김 |
| VAPID, 서비스 워커, 구독 | 새 소식 알림 | 실패 시 원인 대신 복구 행동을 안내 |
| role, RBAC, `admin`, `super_admin` | 운영 권한, 담당 업무별 접근 권한 | 일반 화면에 내부 권한 식별자를 노출하지 않음 |
| RSVP | 행사 신청 | 동문이 익숙한 행동 용어 사용 |

## 오류와 상태 문구

- 일반 동문 화면의 오류는 내부 코드나 객체 문자열 대신 현재 상태와 다음 행동을 안내합니다.
- 알 수 없는 오류는 잠시 후 재시도하도록 안내하고, 계정 문제는 동문회 사무국 문의로 연결합니다.
- 일반 화면에는 VAPID, token, role, RBAC, HTTP 상태나 내부 오류 코드를 표시하지 않습니다.
- 운영자 전용 화면은 업무 수행에 필요한 상세 오류와 운영 용어를 보존할 수 있지만 API 내부 식별자를 그대로 노출하지 않습니다.
- 성공 문구는 완료된 행동을 짧고 분명하게 표현합니다.

## 화면별 기준

- 헤더·Drawer: `동문 가입 신청`, `첫 로그인 설정`, `이용 안내`
- 로그인·가입: 승인 주체는 `동문회 사무국`, 승인 후 행동은 `첫 로그인 설정`
- 문의: `동문회 사무국 문의`와 문의 가능한 범위를 함께 표시
- 알림: `새 소식 알림 켜기/끄기` 중심으로 안내하고 브라우저 내부 구현은 숨김

## 가입부터 첫 로그인까지

가입 여정은 다음 네 단계를 같은 순서와 이름으로 안내합니다.

1. 가입 정보 보내기
2. 동문회 사무국 확인
3. 비밀번호 만들기
4. 첫 로그인 완료

- 가입 신청 직후에는 아직 실행할 수 없는 로그인·비밀번호 만들기 버튼을 노출하지 않습니다.
- 승인 안내 링크로 들어오면 코드는 자동으로 적용하고 비밀번호 입력만 요청합니다.
- 코드 직접 입력은 안내 링크를 사용할 수 없을 때의 보조 경로로 제공합니다.
- 만료되거나 잘못된 링크는 사무국 문의, 이미 사용한 링크는 로그인으로 연결합니다.
- 로그인 화면은 기존 동문, 처음 방문한 동문, 승인 안내를 받은 동문의 경로를 구분합니다.
- 메뉴의 `첫 로그인 설정`은 여정 전체의 이름이고, 3단계 `비밀번호 만들기`는 해당 화면에서 수행하는 구체적인 행동입니다.

## 홈의 정보 우선순위

- 홈은 기능 메뉴를 반복해서 나열하지 않고 `핵심 행동 → 중요 공지·다가오는 행사 → 동문 소식 → 인사말` 순서로 안내합니다.

## 커뮤니티 유형

- `discussion`은 화면에서 `자유게시판`으로 표시하고 작성자와 대화를 먼저 보여 줍니다.
- `question`은 `묻고 답하기`로 표시하고 질문의 상황과 답변 참여를 강조합니다.
- `share`는 `동문 이야기·행사 후기`로 표시하고 인물·사진·현장 기록을 우선합니다.
- `congrats`는 `경조사`로 표시하고 구분·일시·장소·연락 정보를 본문에서 명확히 안내합니다.
- `notice`는 `공지사항`, `news`는 `동문 소식`으로 표시하며 내부 category 값은 사용자에게 노출하지 않습니다.
- 작성자 이름이 없을 때 번호를 조합한 이름을 만들지 않고 `이름을 공개하지 않은 동문`으로 안내합니다.
- 빈 목록, 검색 무결과, 불러오기 실패를 서로 구분하고 각각 첫 글 유도, 검색 초기화, 재시도 행동을 제공합니다.
- 비회원은 `동문 가입 신청·동문 로그인·비밀번호 만들기`, 회원은 `동문 찾기·게시판 보기·내 정보 확인`, 운영자는 `가입 신청·공지/소식·행사 관리`를 핵심 행동으로 봅니다.
- 공지·행사·소식은 화면 크기와 관계없이 같은 순서와 의미를 유지합니다. 작은 화면이라는 이유로 행사나 소식을 숨기지 않습니다.
- 이미지 슬롯은 정해진 비율을 유지하고 실제 사진이 없거나 읽히지 않을 때에도 제목과 날짜를 통해 내용을 이해할 수 있어야 합니다.

## 동문 수첩·행사·내 정보

- 공개 범위는 내부 값(`all`, `cohort`, `private`) 대신 `모든 동문에게 공개`, `같은 기수에게만 공개`, `나만 보기`로 표시하고 실제 노출 영향을 함께 설명합니다.
- 동문이 공개하지 않은 항목은 하이픈이나 빈칸 대신 `공개하지 않음`, `공개된 상세 정보가 아직 없어요`처럼 의도된 상태로 안내합니다.
- 검색 무결과는 조건을 줄이거나 모든 필터를 초기화하는 행동을 제공합니다. 목록 조회 실패는 검색 무결과로 위장하지 않고 다시 불러오기를 제공합니다.
- 행사 신청 기록이 없는 상태와 취소 상태는 `아직 신청하지 않았어요`로 안내합니다. 신청 완료·대기 중·미신청은 현재 상태의 의미와 다음 행동을 함께 표시합니다.
- 정원이 찬 경우 대기 전환은 서버가 결정하며, 사용자가 임의로 `대기` 상태를 선택하게 하지 않습니다.
- 내 정보는 저장 전 변경 여부, 선택한 공개 범위의 영향, 저장 완료 결과를 화면에 남깁니다.
- 정보가 없거나 수정·참여가 어려운 상태에서는 동문회 사무국 문의로 연결합니다.
